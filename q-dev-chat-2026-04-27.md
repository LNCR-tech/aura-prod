what if i used the same exact variables.py for two seed runs but in the second run, at least one of the variables is modified, like turning max students from 200 to 199

---


The entire RNG sequence shifts from that point forward.

Here's why: `random.Random(42)` produces a fixed sequence of numbers — call them `r1, r2, r3, r4, ...`. Every call to `rng.randint()`, `rng.random()`, `rng.choice()` etc. consumes the next number in that sequence in order.

If you change `max_students` from 200 to 199, the student count draw might return the same number (say 185), but if it returns a different number (say 184 instead of 185), then one fewer student is generated — meaning one fewer `rng.choice()` for names, one fewer `rng.randint()` for student ID, one fewer `rng.random()` for suffix, etc. That's 3-4 fewer RNG draws per missing student.

Every subsequent draw — events, attendance, sanctions — is now offset by that many positions in the sequence. So the rest of the data is completely different even though only one variable changed.

This is expected behavior for a seeded RNG — it's only reproducible if the **exact same sequence of calls** is made. Same variables = same data. Any change = divergence from that point forward.

Your categorization is spot on, and yes, the 3rd type definitely exists. Here's how I'd name them:

**Type 1 — Free variables**
You set them to whatever, nothing else cares. Example: `PGADMIN_DEFAULT_EMAIL`, `PGADMIN_DEFAULT_PASSWORD`. Only pgAdmin reads them, no other service references them.

**Type 2 — Required/consumed variables**
The service needs them to function. Example: `POSTGRES_USER`, `POSTGRES_PASSWORD`. Postgres won't start without them.

**Type 3 — Contract variables**
You can set them to whatever, but other services must match. Example: `DB_PORT`, `DB_USER`, `DB_PASSWORD`. If you change `DB_PORT` from `5432` to `5433`, you must also update `DATABASE_URL` in `backend/.env` and `ASSISTANT_DB_URL` in `assistant/.env` to reflect that port. The value is your choice, but the choice propagates.

In `database/.env`, almost everything is Type 3 — because the whole point of the database service is to be consumed by other services. `PGADMIN_*` is the only true Type 1.
